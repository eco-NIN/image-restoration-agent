### 第 1 步：确认现状与边界

- 确认后端 **没有删除历史记录接口**，因此前端无法实现真正“删（Delete）”闭环。

### 第 2 步：后端新增“删除历史”能力（不影响现有接口）
文件：`backend/app/main.py`

#### 2.1 新增安全文件清理工具函数
- `safe_unlink(path_text: str)`
- `safe_rmtree(path: Path)`

作用：
- 删除文件/目录前校验目标路径必须在 `backend/runtime` 下，避免误删非运行目录内容。
- 仅在路径存在且类型正确时执行删除。

#### 2.2 新增删除单条历史接口
- `DELETE /api/history/task/{task_id}`

行为：
- 若任务不存在：返回 404。
- 若任务仍在 `running/cancelling`：返回 409（阻止删除进行中任务）。
- 删除数据库 `tasks` 表对应记录。
- 清理该任务对应 runtime 文件：
  - 输入图片
  - 结果图片
  - `backend/runtime/jobs/{task_id}`

#### 2.3 新增删除批次历史接口
- `DELETE /api/history/group/{group_id}`

行为：
- 按 `upload_group` 查询批次所有任务。
- 若批次不存在：返回 404。
- 若批次内存在进行中任务：返回 409。
- 批量删除数据库记录。
- 清理批次关联 runtime 数据：
  - 每个任务对应 `jobs/{task_id}`
  - `uploads/{group_id}`
  - `results/{group_id}`

说明：
- 这一步实现了数据库层面的“删”与运行时文件同步清理，满足后台管理系统常见数据治理流程。

## 为什么这样设计（见解）
1. **最小侵入原则**
   - 你强调“原有功能不能变”，因此没有改导航结构、页面主布局、任务提交流程，仅在历史页上端叠加管理功能。

2. **CRUD 展示目标对齐**
   - C（Create）：由现有任务提交自然体现。
   - R（Read）：历史列表 + 详情查看 + 前端筛选。
   - U（Update）：任务状态变化（处理中/取消中/完成）与取消动作仍保留。
   - D（Delete）：新增单条/批次删除与批量删除，形成完整闭环。

3. **数据库存储能力可展示**
   - 删除操作已落到 SQLite 的 `tasks` 表，且与 runtime 文件同步清理，能在毕设答辩中说明“数据库记录与文件资源一致性维护”。

4. **安全性考量**
   - 删除文件前增加 runtime 目录边界校验，避免误删其他路径。
   - 对进行中任务返回 409，避免删除并发状态造成脏数据。
