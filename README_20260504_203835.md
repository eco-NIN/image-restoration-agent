# 多轮 Plan 工作流可视化重构记录（2026-05-04 20:38:35）

## 本次目标
根据你反馈的问题，按 4KAgent 的真实 `workflow.log` 语义重构前后端流程展示，重点解决：
- 多轮计划（Plan/Adjusted plan）杂糅、顺序错乱
- 执行阶段混入进度条乱码（Decoder Task Queue）
- 反馈阶段“过早完成”（仅出现 Best tool 就显示完成）
- 打分与选择覆盖前一次，无法分轮理解

## 关键结论（结合 4KAgent 工作流）
4KAgent 的真实流程是：
1. 初始 `Plan`
2. 按序执行子任务并为每一步试跑多工具
3. 若失败触发 `Back to ...` + `Roll back for ...`
4. `Adjusted plan` 重新规划
5. 继续执行直到 `Restoration result`

因此前端不能再用“单层列表”展示，而应改为“按 attempt 分轮次时间线”。

## 改动日志（逐步）

### Step 1. 后端 flow 解析改为严格基于 workflow.log
- 文件：`backend/app/main.py`
- 修改：
  - `parse_flow_from_task` 入参由 `merged_logs` 改为 `workflow_logs`。
  - `get_task_status` 中 `flow = parse_flow_from_task(task_id, workflow_tail)`。
- 原因：`db_logs` 中包含 stdout 进度条等噪声，不适合做流程可视化解析。

### Step 2. 新增 attempt（轮次）时间线模型
- 文件：`backend/app/main.py`
- 修改：
  - 在 `parse_flow_from_task` 中新增 `attempts` 结构。
  - 遇到 `Plan:` 或 `Adjusted plan:` 时启动新 attempt。
  - 遇到 `Executing ... on ...` 时在当前 attempt 下创建 step。
  - step 内记录：`subtask`、`executionTarget`、`toolTrials`、`bestTool`、`qualityScore`、`faceScores`。
- 效果：每一轮计划和对应执行步骤被清晰分组，不再混在同一层。

### Step 3. 过滤执行乱码（Decoder Task Queue）
- 文件：`backend/app/main.py`
- 修改：
  - 新增 `is_runtime_noise`，过滤典型 tqdm/progress-bar 噪声：
    - `Decoder Task Queue:`
    - 含 `xx%|` 进度片段
    - `| n/m [` 进度格式
- 效果：执行面板不再被进度条刷屏污染。

### Step 4. 反馈阶段完成条件修正
- 文件：`backend/app/main.py`
- 修改：
  - `feedback.done` 由 `bool(best_tool or final_result)` 改为 `bool(final_result)`。
- 效果：只有出现 `Restoration result:` 才标记“反馈已完成”，与真实流程一致。

### Step 5. 前端实时流程改为按 attempt 渲染
- 文件：`frontend/src/pages/WorkbenchPage.jsx`
- 修改：
  - 新增状态样式工具：`attemptStatusLabel`、`attemptStatusStyle`、`prettyTimestamp`。
  - `mergeFlowProgress` 支持合并 `attempts` 和 `scoreLines`。
  - “实时流程可视化”区域新增“按工作流分轮次（Attempt）”卡片：
    - 每轮显示：轮次号、初始/重规划标识、状态、开始时间、回滚说明、该轮计划
    - 每 step 显示：执行输入节点、工具试跑列表、人脸评分、最优工具、质量分、结果行
- 效果：展示顺序严格对应 `workflow.log` 的时间线，用户可按轮次理解整个过程。

### Step 6. 打分区语义修正
- 文件：`frontend/src/pages/WorkbenchPage.jsx`
- 修改：
  - “打分与选择”标题改为“打分与选择（最终汇总）”。
  - 新增说明：分轮细节请看 attempt 卡片中的工具试跑与质量分。
- 效果：避免“只看到最终文件，误以为覆盖了前一次”的歧义。

## 我的见解
这类问题本质不是单纯 UI 样式，而是“事件模型不对”。
- 原模型：平铺字段（planList/executionSteps/scoreLines）
- 新模型：事件流 -> attempt -> step

对于 Agent 系统，这种“轮次化 + 时间线”模型是必要的，因为重规划与回滚是常态，不是异常。
