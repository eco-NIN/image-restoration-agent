# 多 Plan 实时流程可视化修复记录（2026-05-04 19:43:41）

## 任务背景
- 问题现象：工作台“实时流程可视化”在单一超分 plan（例如仅 `['super-resolution']`）时基本正常；在多 plan（例如 `['jpeg compression artifact removal', 'defocus deblurring', 'super-resolution', 'denoising']`）时出现显示混乱。
- 典型错误：阶段条出现“已到执行完成后又跳回感知/决策，且前序完成态丢失”的错乱。
- 约束：`4KAgent` 目录保持不变（本次未改动该目录）。

## 根因分析（结合你给出的日志）
1. `workflow.log` 在多 plan 场景会更长，且存在“多轮计划”。
- 先出现 `Plan: [...]`，执行一段后可能出现 `Adjusted plan: [...]`。
- 旧逻辑只取第一次 `Plan`，且没有历史版本概念，导致决策面板对多轮计划表达能力不足。

2. 执行阶段识别规则过窄。
- 旧逻辑仅识别 `Executing ... on input...`。
- 多 plan 的后续步骤是 `Executing ... on <previous_subtask>@<tool>...`，因此会漏记执行步骤，进而让“执行完成”状态被错误判断。

3. 日志读取窗口过小且阶段状态无记忆。
- 任务状态接口只读取 `workflow.log` 尾部 120 行。
- 多 plan 长任务中，早期 `IQA / Image description / Plan` 行会被挤出窗口。
- 当这些行临时不可见时，前端接收到的 `flow.stages` 会表现为前序阶段“未完成”，造成你看到的“回退和错乱”。

## 改动日志（逐步）

### Step 1: 扩大 workflow 日志窗口，降低阶段信息丢失风险
- 文件：`backend/app/main.py`
- 修改点：
  - `read_workflow_log_tail(task_id, max_lines=120)` -> 默认 `2000`
  - `get_task_status` 中调用由 `max_lines=120` -> `max_lines=2000`
- 目的：减少多 plan 长日志时早期阶段信息被尾部截断的概率。

### Step 2: 后端 flow 解析支持多轮计划（Plan + Adjusted plan）
- 文件：`backend/app/main.py`
- 修改点：
  - 新增 `adjusted_plan_text`。
  - 新增 `parse_plan_text_to_list(raw_plan)` 统一解析计划列表。
  - 解析 `Plan:` 时不再只取第一次，改为持续更新最新计划。
  - 新增解析 `Adjusted plan:`，并将其作为更高优先级决策详情。
  - 新增 `planHistory`（去重后保留计划演进快照）。
- 目的：多 plan/重排计划场景下，决策展示不再覆盖错乱，而是反映演进过程。

### Step 3: 执行阶段识别从“仅 input”放宽为“任意 Executing”
- 文件：`backend/app/main.py`
- 修改点：
  - `elif "Executing" in clean and "on input" in clean` -> `elif "Executing" in clean`
  - 子任务名提取从 `split("on input", 1)` -> `split("on", 1)`。
- 目的：捕获多 plan 后续链路的执行节点，保证执行阶段进度连续。

### Step 4: 前端为 flow 增加“单向完成态合并”，防止阶段回退闪烁
- 文件：`frontend/src/pages/WorkbenchPage.jsx`
- 修改点：
  - 新增 `mergeFlowProgress(prevFlow, nextFlow)`。
  - 在 `normalizeJob` 中将 `flow` 设置为 `mergeFlowProgress(prevJob.flow, taskData.flow)`。
  - `stages.done` 采用 OR 合并（新旧任一为真即保持完成）。
  - 同步合并 `planHistory` 与 `planList`，避免轮询瞬时空值覆盖已有信息。
- 目的：即使某次轮询的解析数据短暂不完整，也不会把已完成阶段重置为未完成。

### Step 5: 前端决策区展示多 plan 历史
- 文件：`frontend/src/pages/WorkbenchPage.jsx`
- 修改点：
  - 当 `flow.planHistory.length > 1` 时，展示“计划版本 N / 当前计划”块。
  - 仍保留当前 `planList` 的统一展示。
- 目的：让多 plan 场景结构更清晰，不再“看起来乱”。

## 我的见解
- 这类可视化错乱通常不是“UI 样式问题”，而是“日志采样窗口 + 增量状态机设计”的组合问题。
- 你的场景本质是“流式、可回溯、可修正的 Agent 过程”，因此建议遵循两条原则：
  - 阶段状态单向推进（monotonic progress）
  - 计划与执行支持版本化（history-aware rendering）
- 本次修复已按这两条原则落地，能够覆盖你提到的多 plan 主路径问题。
